# slora

A secure LoRa mesh protocol and its OMNeT++ simulator. The protocol
stack is pure Zig; the simulator wraps it in a C++ OMNeT++/INET/FLoRa
harness. Everything (OMNeT++, INET, FLoRa) lives in a podman image to
sidestep host-glibc / Qt issues.

## Layout

```
src/
├── stack/      Pure Zig protocol stack. No I/O. Tested with zig build test.
│   └── src/      crypto, types, schedule, link, route, relay, announce,
│                 tsch, echo, slora (node state), c_abi (C ABI shim)
├── sim/        OMNeT++ C++ harness around the Zig stack.
│   ├── SloraApp.{cc,h}    per-node app; drives the radio, calls into Zig
│   ├── slora.{h}          C ABI exposed by libslora.a
│   ├── SloraNode.ned      two-radio composition per node
│   └── SloraMesh.ned      top-level network + LoRa medium
└── flood/      Three flood baselines: FloodSNR (Meshtastic-style),
                FloodNaive (always rebroadcast), FloodMPR (OLSR-style).

simulations/
└── omnetpp.ini       default smoke/example configs

study/
├── generate.py       writes simulations/study.ini, config_list.txt, topology_specs.json
├── run_all.sh        runs every config in the container; resumable
├── run_parallel.sh   parallel variant
└── analyze.py        parses .sca → results.csv + plots + topology renders

Dockerfile          builds the slora-sim podman image
justfile            host-side commands: build, shell, run, study, clean
Makefile            forwards `make` into the container
```

## Pinned versions

| component | version |
|---|---|
| Ubuntu | 26.04 |
| OMNeT++ | 6.3.0 |
| INET | 4.5.4 (symlinked as `inet4.4` so FLoRa resolves) |
| FLoRa | 1.2.0 |

Bump together at the top of `Dockerfile`.

## Build the image

```sh
direnv allow
just build         # ~20 min the first time
```

## Run a simulation

```sh
just run                   # smoke config
just run FloraSmall        # named config from simulations/omnetpp.ini
just shell                 # interactive shell in the container
```

Inside the container (`/work` is bind-mounted to the project root):

```sh
make run                       # builds, then runs config Smoke
make CONFIG=Foo run            # any config
make clean                     # nuke build artifacts
```

## Reproduce the 20,000-run study

```sh
just shell bash -c "python3 /work/study/generate.py"     # rewrite study.ini
just shell bash /work/study/run_parallel.sh              # all 20,000 runs
just shell bash -c "python3 /work/study/analyze.py"      # results.csv + plots
```

`run_*.sh` is resumable: it skips any `.sca` already present in
`simulations/results/`. To force a fresh run, delete those first.
OMNeT++ appends to existing `.sca` files instead of overwriting them.

## Read the code (suggested order)

1. `src/sim/SloraMesh.ned`: the network shape (NED is the declarative layer).
2. `src/sim/SloraNode.ned`: one node, with 2 LoRa NICs, mobility, and the app.
3. `src/sim/SloraApp.cc`: the C++ glue. Calls into Zig via the C ABI.
4. `src/sim/slora.h`: the C ABI surface, generated by `zig build`.
5. `src/stack/src/slora.zig`: the Zig entry point. `Node` state machine.
6. The per-layer Zig files (`schedule.zig`, `tsch.zig`, `link.zig`,
   `route.zig`, etc.).

Then `src/flood/FloodApp.cc` for the baseline. Much simpler, useful contrast.

## Crypto

Ed25519 sigs, X25519 ECDH, ChaCha20-Poly1305 AEAD, BLAKE2b for address
hashing. All from `std.crypto`. No external dependency.

## AI usage

I used <https://ai.omnetpp.org/>, the official OMNeT++ AI documentation
site, while learning the OMNeT++ / INET / FLoRa APIs that the C++
harness in `src/sim/` and `src/flood/` builds on.
